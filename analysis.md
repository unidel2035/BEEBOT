# BEEBOT — Анализ текущего состояния

> **Дата:** 29 марта 2026
> **Аудит:** Claude Code (Sonnet 4.6)
> **Масштаб:** полный — 5 500+ строк Python-ядра, 14 Vue-страниц, 284 теста, 5 внешних API, DEVBOT-агент разработчика

---

## 1. Общая оценка

BEEBOT — производственная CRM-связанная система с многоагентным AI-ядром. За ~5 недель разработки прошла 7 фаз. На 29.03.2026 работает в продакшне (@AleksandrDmitrov_BEEBOT).

Ключевой факт: **система эволюционировала** от простого бота (Фаза 0) до полноценной платформы:
- Telegram-бот с 5 специализированными агентами
- Веб-панель (PWA, 14 страниц)
- DEVBOT — автономный разработчик (реализован в Фазе 7)
- Режим работника склада (Фаза PR-29.03)
- LLM-ассистент пчеловода с CRM-снимком

**CRM на момент анализа:** 76 товаров · 285+ клиентов · 382+ заказов · 230+ UDS-транзакций

## 4. Слабые стороны и технический долг

### 4.1 Архитектурные проблемы

#### ⚠️ bot.py — 1 899 строк (монолит)

Файл `src/bot.py` вырос до 1 899 строк и содержит:
- Хэндлеры всех пользовательских ролей (подписчик, работник, администратор)
- FSM оформления заказа (LogistAgent FSM hooks)
- FSM Инспектора (InspectFSM hooks)
- Startup/shutdown, polling, фоновые задачи
- 3 режима AdminChatAgent (клиент / работник / назад)

Это нарушение принципа единственной ответственности. Риск при добавлении новых фич — коллизии хэндлеров, сложность отладки.

**Рекомендация:** разбить на aiogram Router-модули: `routers/user.py`, `routers/worker.py`, `routers/admin.py`, `routers/fsm_order.py`

#### ⚠️ Дублирование логики AdminChatAgent

`admin_chat.py` содержит два почти идентичных метода:
- `_get_crm_context()` — live-запросы к CRM (~170 строк)
- `_build_context_from_snapshot()` — из CrmSnapshot (~160 строк)

Одинаковая логика построения помесячной/подневной статистики, форматирования заказов, товаров, клиентов. Если бизнес-правила изменятся — нужно обновлять в двух местах.

**Рекомендация:** единый `_build_context(orders, products, clients)` → вызывается из обоих путей.

#### ⚠️ Инспектор вне оркестратора

`InspectorAgent` доступен только через команду `/inspect`. Оркестратор не знает об интенте «осмотр улья». Если пользователь напишет «осмотри мой улей» — запрос попадёт в `consult` к консультанту, а не к инспектору.

Схема интентов оркестратора: `consult | order | edit | track | stats | greeting` — нет `inspect`.

**Рекомендация:** добавить `inspect` в `_INTENT_SYSTEM` и граф LangGraph.

#### ⚠️ BEEKEEPER_CHAT_ID vs ADMIN_IDS — путаница ролей

В `config.py`:
- `BEEKEEPER_CHAT_ID` — для уведомлений (Notifier)
- `ADMIN_CHAT_ID` — для /admin, /stats, /faq команд
- Fallback: если ADMIN_CHAT_ID не задан → берётся BEEKEEPER_CHAT_ID

При мультиадминной настройке (`ADMIN_CHAT_ID=123,456`) только первый ID становится `ADMIN_CHAT_ID`. При этом `ADMIN_IDS = frozenset(все)`, но `ADMIN_CHAT_ID = первый`. Путаница при расширении команды.

#### ⚠️ Память работника в RAM

`worker.py`: чеклист сборки `_checked: dict[tuple[int, int], set[int]]` хранится в памяти процесса. При рестарте бота:
- Заказ остаётся в статусе «В сборке» в CRM
- Чеклист пуст — работник не знает что отмечено
- Нет автоматической синхронизации при старте

### 4.2 Логические конфликты

#### ⚠️ UDS сопоставляет товары по имени, а не по SKU

В `uds.py:_build_order_items()`:
```python
product = await integram_client.get_product_by_name(good["name"])
```
UDS-транзакция содержит `sku` (externalId товара), но поиск ведётся по имени. Если название товара в UDS отличается от Integram (регистр, пробелы, артикул) — создаётся позиция с `product_id=0`. Это причина частичного заполнения состава заказов.

#### ⚠️ Worker notify получает _worker_notifier через глобальный импорт

`uds.py:sync_uds_transaction()`:
```python
from src.notifications import _worker_notifier
if _worker_notifier:
    await _worker_notifier.notify_workers_new_order(...)
```
Глобальная переменная инициализируется в `bot.py` при старте. Если порядок инициализации нарушится — уведомления работникам молча не отправятся (нет исключения, только `if _worker_notifier:`).

#### ⚠️ DEVBOT → BEEBOT через localhost:8091 без авторизации

Команда `/dev` в `bot.py` → `httpx.post(DEVBOT_API_URL + "/task")`. DEVBOT FastAPI запущен с `allow_origins=["*"]` и без авторизации. Если порт 8091 доступен снаружи (VPS) — любой может поставить задачу DEVBOT.

При нормальной конфигурации (DEVBOT на hive, BEEBOT на VPS через SSH-туннель) этого не происходит, но риск есть при изменении деплоя.

#### ⚠️ extract_fact() — regex без учёта контекста

`memory.py:extract_fact()` использует простые regex-паттерны:
```python
(re.compile(r'у меня (язва|гастрит|диабет|...)', ...), "health")
```
Если пользователь пишет «у меня нет язвы» — факт «у меня нет язвы» всё равно будет извлечён и сохранён как health-факт. Нет нейтрализации отрицаний.

#### ⚠️ Позиции заказов: историческая проблема данных

В CRM только ~32 позиции на 382+ заказов:
- TG-заказы до 23.03.2026 — без позиций (старый логист их не создавал)
- UDS-заказы — `product_id=0` из-за несовпадения имён товаров
- AdminChatAgent честно сообщает «нет позиций в CRM» — это данные, не баг

**WorkerAgent** показывает «позиции загружаются...» для заказов без позиций — пустой список подтянется из CRM, интерфейс не сломается.

### 4.3 Инфраструктурные риски

| Риск | Описание | Вероятность | Влияние |
|------|----------|-------------|---------|
| Обрыв SSH-туннеля hive→VPS | Groq API недоступен — консультант не отвечает | Средняя | Высокое |
| Падение hive | SPOF: Groq-прокси + Telegram SOCKS5 + DEVBOT — всё на hive | Низкая | Критическое |
| OOM на VPS | beebot 762 MiB + beebot-web 76 MiB + soundmeter + buddy + traderagent = ~1.19 GiB из 1.921 GiB лимита | Средняя (со swap) | Высокое |
| UDS catch-up при рестарте | При холодном старте catch-up с 01.01.2024 — нагрузка на UDS API ~3-5 мин | Каждый рестарт | Низкое |
| DEVBOT MemoryStorage | FSM состояния DEVBOT в памяти — потеря при рестарте devbot.service | Средняя | Среднее |
