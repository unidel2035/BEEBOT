# BEEBOT — Анализ проекта

> **Дата:** 2 апреля 2026
> **Предыдущий анализ:** 29 марта 2026

---

## 1. Общее состояние

BEEBOT — production-система: Telegram-бот + веб-панель для управления заказами пчелопродуктов. Работает с реальными клиентами и заказами.

| Метрика | Значение |
|---------|----------|
| Python-файлов в src/ | 102 |
| Строк кода (src/) | ~22 000 |
| Тестов | 354 (31 файл) |
| Товаров в CRM | 85 (75 с UDS-маппингом) |
| Клиентов | ~2 800 |
| Заказов | ~2 000 |
| Docker-контейнеров | 3 (redis + beebot + beebot-web) |

---

## 2. Сильные стороны

### Функциональность
- **Полный цикл заказа**: от консультации до доставки с автотрекингом
- **5 агентов LangGraph**: консультант, логист, аналитик, инспектор, ассистент
- **Гибридный поиск KB**: 70% семантика + 30% стилометрия + keyword-буст + онтология (74 симптома → 77 показаний)
- **UDS-интеграция**: импорт заказов из магазина с составом через Admin API, маппинг по sku_uds
- **PWA веб-панель**: 14 страниц, offline-режим, SSE-уведомления
- **DEVBOT**: автономный разработчик на Claude API

### Инфраструктура
- **CI/CD**: GitHub Actions (lint + test + deploy)
- **Docker**: изолированные контейнеры, healthcheck
- **Обходные туннели**: Groq и Telegram API через SSH
- **Резервное копирование**: Яндекс Диск (код готов, токен получен)

### Архитектурный фундамент (steps 0–5)
- **Redis Streams**: EventBus для межпроцессного общения
- **Service Layer**: OrderService + NotificationService
- **BusHandlers**: Backend слушает шину
- **BotServiceClient**: тонкий клиент для бота
- **BackgroundTaskManager**: управляемые фоновые задачи

## 3. Критические проблемы

### 3.1 Незавершённый рефакторинг — мёртвый код

Steps 0–5 создали новую структуру (infrastructure/, transport/, domain/, services/), но **ни один файл не импортируется из новых путей**. 28 файлов — точные копии оригиналов.

| Новый слой | Файлов-копий | Импортов из production | Статус |
|---|---|---|---|
| src/infrastructure/ | 18 | 0 | Мёртвый код |
| src/transport/ | 6 | 0 | Мёртвый код |
| src/domain/ | 4 | 0 | Мёртвый код |
| src/services/ | 3 | 0 | Мёртвый код |

**Решение:** Удалить копии. services/ и bus.py оставить — они содержат новый код.

### 3.2 Три пути создания заказов

OrderService создан, но **не подключён**:

| Путь | Файл | Через OrderService? |
|---|---|---|
| Telegram-бот | agents/logist.py | Нет — прямой CRM |
| Веб-панель | web/routers/orders.py | Нет — прямой CRM |
| UDS | integrations/uds.py | Нет — прямой CRM |

### 3.3 Три системы уведомлений

| Система | Файл | Используется? |
|---|---|---|
| Notifier (aiogram) | notifications.py | Да — бот |
| HTTP-функции (httpx) | web/notifications.py | Да — веб |
| NotificationService | services/notification_service.py | Нет |

### 3.4 CRM-доступ из 17 мест

IntegramClient импортируется напрямую из 17 файлов. Нет единой точки входа.

### 3.5 Чеклист работника в RAM

При рестарте бота чеклист теряется. SharedContext имеет поле `checklist`, но не используется.

## 4. Средние проблемы

- **AGENT_SPECS**: TABLE_AGENT_SPECS = None, таблица не создана
- **Gift Protocol**: работает для чтения, обходится при создании заказов
- **Integram**: 6 битых страниц после массового обновления (обход через try/except)
- **Дата Integram**: формат записи MM/DD/YYYY ≠ формат чтения DD.MM.YYYY (парсер обновлён, не закоммичен)
- **Клиенты**: 96% без телефона (UDS-импорт)
