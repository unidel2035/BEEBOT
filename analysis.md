# BEEBOT — Анализ текущего состояния проекта

> Дата: 18 марта 2026
> Автор анализа: Claude Code (Opus 4.6)
> Масштаб: полный аудит всех подсистем (8530 строк Python, 11 Vue-страниц, 284 теста)

## 1. Общая оценка

Проект BEEBOT прошёл путь от простого FAQ-бота до многокомпонентной production-системы. За 2 недели активной разработки реализованы:

- Многоагентная архитектура (LangGraph) с 3 агентами и 6 типами интентов
- Полная CRM-интеграция (Integram) с CRUD для товаров, заказов, клиентов
- PWA веб-панель с offline-режимом (сборка + склад)
- Доставка: расчёт стоимости (СДЭК, Почта России), авто-трекинг, уведомления
- Система тестов: 284 теста + CI/CD через GitHub Actions

**Текущий масштаб:**
- ~8530 строк Python (без тестов), ~5000 строк тестов
- 11 Vue-страниц, 2 PWA-терминала
- 2 Docker-контейнера (бот + веб)
- CRM: 76 товаров, 285+ клиентов, 326+ заказов

**Статус по фазам плана:**
- Фаза 0 (Стабилизация): ✅ завершена
- Фаза 1 (Качество кода): ✅ завершена
- Фаза 2 (Функциональность): ✅ завершена
- Фаза 3 (Масштабирование): 🔜 не начата
- Фаза 4 (Инфраструктура): 🔜 не начата

## 2. Сильные стороны

### 2.1 Функциональные

| Аспект | Описание |
|--------|----------|
| **Гибридный поиск** | Уникальная комбинация семантики (70%) + стилометрии (30%) + keyword-буст — точные ответы в стиле автора |
| **Многоагентная архитектура** | LangGraph с 6 типами интентов: consult, order, delivery, status, stats, chitchat |

| **Пагинация + поиск** | Серверная пагинация всех списков, текстовый поиск, фильтры по статусу/источнику |
| **Экспорт CSV** | Заказы, клиенты, товары — utf-8-sig для Excel |
| **SSE-уведомления** | Real-time обновления в веб-панели (смена статуса, трекинг) |
| **История диалога** | Консультант помнит последние 5 пар сообщений (TTL 30 мин) |
| **Валидация телефона** | Нормализация +7/8/9xx → +7XXXXXXXXXX с подсказками ошибок |

## 3. Критические проблемы

### 3.1 Отсутствует эндпоинт создания заказа через веб-панель

**Файл:** `src/web/api.py`
**Проблема:** Страница `NewOrderView.vue` отправляет `POST /api/orders`, но такого эндпоинта нет в API. Попытка создать заказ через веб-панель приводит к **404**.
**Влияние:** Функционал «Создание заказа» в веб-панели полностью неработоспособен.
**Решение:** Реализовать `POST /api/orders` в api.py — создание заказа, позиций, подсчёт суммы.

### 3.2 Фронтенд не использует пагинацию

**Файлы:** `web/src/views/OrdersView.vue`, `ClientsView.vue`, `ProductsView.vue`
**Проблема:** После добавления серверной пагинации (API возвращает `{items, total, page, pages}`) фронтенд по-прежнему ожидает массив. Результат — пустые списки или ошибки.
**Влияние:** Все три страницы списков потенциально сломаны.
**Решение:** Обновить фронтенд: `response.data.items` вместо `response.data`, добавить компонент пагинации.

### 3.3 Формат дат: бэкенд vs фронтенд

**Файлы:** `src/web/api.py` → `web/src/utils.js:formatDate()`
**Проблема:** Бэкенд возвращает даты в формате `"DD.MM.YYYY"` (строка), а `formatDate()` ожидает ISO-формат или `Date` объект. Все даты в интерфейсе отображаются как `Invalid Date` или `undefined`.
**Решение:** Обновить `formatDate()` для парсинга `DD.MM.YYYY`, или отдавать ISO-8601 с бэкенда.

### 3.4 Четыре test-файла с ошибками сборки

**Файлы:** `tests/test_integration.py`, `tests/test_knowledge_base.py`, `tests/test_orchestrator.py` + 1
**Проблема:** 4 ошибки при `pytest --collect-only`. Тесты собираются, но не запускаются.
**Влияние:** 284 теста собрано, но ~40 не могут запуститься из-за ошибок импорта/зависимостей.
**Решение:** Исправить импорты и мок-зависимости в проблемных тестах.

## 6. Конфликты и расхождения логик

### 6.1 Два CRM-клиента — разные интерфейсы

```
integram_api.py (IntegramAPI)         integram_client.py (IntegramClient)
├── get_all_objects(type_id)           ├── get_orders(status?, limit?)
├── create_object(type_id)             ├── get_products()
├── set_requisites(obj_id, data)       ├── create_order(client_id, items)
└── Низкоуровневый HTTP                └── Бизнес-логика с маппингом

Проблема: IntegramClient использует IntegramAPI внутри, но web/api.py
использует _get_crm() → IntegramClient. Бот использует integram_client
напрямую. Два уровня абстракции, но маппинги полей отличаются.
```

### 6.2 Статусы заказов — 3 источника истины

```
integram_api.py:    STATUS_IDS = {"Новый": 247, "Подтверждён": 248, ...}
integram_client.py: STATUS_LOOKUP = {"Новый": 247, "Подтверждён": 248, ...}
web/api.py:         ORDER_STATUSES = ["Новый", "Подтверждён", ...]
crm_schema.py:      REQ_STATUS = 2151 (ID реквизита, не значений)

Риск: При добавлении статуса "Возврат" нужно менять 3+ файла.
Уже есть расхождение: crm_constants.py создан, но не все модули его используют.
```

### 6.3 Даты — 4 формата в системе

```
CRM (Integram):  "18.03.2026" (DD.MM.YYYY строка), иногда "31.03.2500"
Бот (Python):    datetime объект или "18.03.2026"
Web API:         "18.03.2026" в JSON (строка)
Frontend:        formatDate() ожидает Date или ISO-8601

Результат: даты в веб-панели показываются как "Invalid Date"
```

### 6.4 Каталог товаров — бот vs веб

```
Бот (logist.py):     Получает каталог через self._crm.get_products()
                     Фильтрует по in_stock=True, группирует по категориям
Веб (api.py):        Получает тот же каталог, но через _get_crm().get_products()
                     Нет фильтрации по in_stock
Различие: Бот показывает только товары в наличии, веб — все
```

### 6.5 Уведомления — два канала

```
bot.py:               notify_admin() — пчеловоду в Telegram при новом заказе
web/notifications.py: notify_client_status_change() — клиенту при смене статуса
web/api.py:           push_event() — SSE в веб-панель при смене статуса

Проблема: Изменение статуса через бота НЕ отправляет SSE в веб-панель.
Изменение через веб-панель НЕ уведомляет пчеловода в Telegram.
Два канала не синхронизированы.
```
