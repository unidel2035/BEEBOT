### 2.1 Функциональные

| Аспект | Описание | Оценка |

### 2.2 Технические

| Аспект | Описание |
|--------|----------|
| **Auto re-auth** | IntegramAPI переаутентифицируется автоматически при 401/403 |
| **Retry + backoff** | 3 попытки с экспоненциальным ожиданием на всех внешних вызовах |
| **Rate limiting** | slowapi: 60 запросов/мин на веб-API |
| **JWT-авторизация** | WEB_PASSWORD обязателен при старте (fail-fast), WEB_SECRET генерируется |
| **CORS** | Ограничен до VPS IP + localhost через WEB_CORS_ORIGINS |
| **Lazy-loading KB** | fastembed-модель загружается при старте контейнера, не при первом запросе |
| **JSON-логирование** | Структурированные логи для мониторинга (time/level/logger/message) |
| **CI/CD** | GitHub Actions: lint (ruff) + 284 теста + деплой при мерже в main |
| **CSV-экспорт** | utf-8-sig кодировка для корректной работы с Excel |
| **Серверная пагинация** | Все списки: page/per_page + поиск + фильтры по статусу/источнику |
| **CRM-снимок** | AdminChatAgent собирает полный снимок CRM: заказы + позиции + топ + клиенты + склад |

## 3. Слабые стороны и технический долг

### 3.1 Архитектурные проблемы

#### ⚠️ Монолитный web/api.py (1 384 строки)
**Файл:** `src/web/api.py`

Весь REST API в одном файле: JWT, dashboard, заказы, клиенты, товары, пользователи, CSV, SSE, healthcheck. Тяжело навигировать, добавлять новые эндпоинты, тестировать изолированно.

**Рекомендация — разбить на blueprints:**
```
src/web/
├── api.py           → main router + startup
├── api_auth.py      → /api/login, /api/users
├── api_orders.py    → /api/orders/*
├── api_clients.py   → /api/clients/*
├── api_products.py  → /api/products/*
├── api_dashboard.py → /api/dashboard/*
└── api_sse.py       → SSE events
```

#### ⚠️ Два CRM-клиента с разными интерфейсами
**Файлы:** `src/integram_api.py` + `src/integram_client.py`

Двухуровневая архитектура оправдана (HTTP-клиент + бизнес-обёртка), но `integram_api.py` содержит таблицы и реквизиты (строки 41–97), которые дублируют `crm_constants.py`.

#### ⚠️ Константы CRM в трёх файлах

| Файл | Что хранит |
|------|-----------|
| `crm_constants.py` | STATUS_IDS, SOURCE_IDS, CATEGORY_IDS — ID-маппинг ✅ |
| `crm_schema.py` | PRODUCT_CATEGORIES, ORDER_STATUSES — строки-списки ⚠️ |
| `integram_api.py` | TABLE_*, REQ_* — ID таблиц и полей ⚠️ |

Разные уровни одного домена. Риск рассинхронизации при изменении структуры CRM.

#### ⚠️ Отсутствие постоянного хранения диалогов

**Файлы:** `src/orchestrator.py` (строки 193–195), `src/agents/admin_chat.py`

Все диалоговые состояния и история хранятся в памяти. При перезапуске бота:
- Теряется контекст всех активных разговоров
- Теряется история AdminChatAgent (история диалога с пчеловодом)
- FAQ-счётчик сбрасывается (если не успел сохранить на диск)

### 3.2 Логические конфликты

#### 🔴 Два независимых пути уведомлений

Клиент и пчеловод уведомляются через **разные механизмы** в зависимости от источника смены статуса:

| Событие | SSE в браузер | TG клиенту | TG пчеловоду |
|---------|:---:|:---:|:---:|
| Смена статуса через **веб** | ✅ `web/api.py` | ✅ `web/notifications.py` | ✅ `web/notifications.py` |
| Смена статуса через **Telegram** `/status` | ❌ | ✅ `src/notifications.py` (Notifier) | ❌ |
| **Авто-трекинг** (tracker.py) | ❌ | ✅ `src/notifications.py` | ❌ |
| Новый заказ через бота | N/A | ✅ | ✅ |
| Новый заказ через UDS | N/A | N/A | ✅ |

**Конкретный конфликт:** `src/notifications.py` (Notifier) и `src/web/notifications.py` — два разных модуля с похожей, но независимой логикой. Разные шаблоны сообщений, разные retry-стратегии.

**Последствие:** Смена статуса через TG-команду `/status` не уведомляет пчеловода и не посылает SSE.

#### ⚠️ Три независимые точки смены статуса

```
1. Telegram /status <id> <статус>  → admin.py    (без SSE, без TG-пчеловоду)
2. PATCH /api/orders/{id}/status   → web/api.py  (с SSE, с TG-клиенту, с TG-пчеловоду)
3. tracker.py (авто)               → tracker.py  (с TG-клиенту, без SSE, без TG-пчеловоду)
```

**Рекомендация:** Единая функция `update_status_and_notify(order_id, new_status, source)` во всех трёх точках.

#### ⚠️ Позиции заказов не сохранены для большинства заказов

В CRM существует только 32 записи позиций на 358 заказов:
- **TG-заказы** (до 23.03.2026) — созданы без позиций (старая версия бота)
- **UDS-заказы** — позиции без привязки к конкретным товарам (UDS отдаёт только итоговую сумму)
- Только 1 ранний заказ (id=1143) имеет полные позиции с именами товаров

AdminChatAgent показывает «нет позиций в CRM» — это отражает реальность, не ошибку кода.

#### ⚠️ Пагинация на фронтенде не используется
**Файл:** `web/src/api.js:79, 139, 154`

Запрашивается `per_page=1000` для заказов, клиентов и товаров, хотя бэкенд поддерживает серверную пагинацию. При 1 000+ заказах будет деградация.

#### ⚠️ tracker.py использует устаревший порт
**Файл:** `src/delivery/tracker.py:26`

`_WEB_INTERNAL_URL` по умолчанию `localhost:8080`, но beebot-web работает на `8088`. Переменная `WEB_INTERNAL_URL` поддерживается через env, но значение по умолчанию неверное.

#### ⚠️ Дублирование профилей клиентов

Клиент, купивший через UDS (telegram_id=0), и тот же клиент в боте (с реальным telegram_id) — два разных профиля в CRM. Механизма объединения нет. Матчинг только по телефону.

### 3.3 Инфраструктурные риски

| Риск | Описание | Вероятность |
|------|----------|-------------|
| Обрыв SSH-туннеля hive→VPS | Groq API недоступен, бот не отвечает на вопросы | Средняя |
| OOM на VPS | 2 GB RAM + 2 GB swap, ~1.66 GB суммарно | Низкая (с swap) |
| Отключение hive | Single point of failure для LLM и Telegram | Высокая |
| JWT инвалидируется при рестарте | WEB_SECRET генерируется заново если не задан в .env | Средняя |
