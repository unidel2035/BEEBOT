# BEEBOT — Анализ текущего состояния проекта

> Дата: 16 марта 2026
> Автор анализа: Claude Code (Opus 4.6)

---

## 1. Общая оценка

Проект BEEBOT прошёл путь от простого FAQ-бота до многокомпонентной системы с CRM-интеграцией, оркестратором на LangGraph, веб-панелью управления и PWA-терминалами для пасеки. Архитектура амбициозна и в целом грамотно спроектирована, но быстрый рост привёл к ряду технических долгов и рассогласованностей между модулями.

**Текущий масштаб:**
- ~15 модулей Python (бот, агенты, CRM, доставка, веб)
- Веб-панель на Vue 3 + PrimeVue (11 views, PWA)
- 2 Docker-контейнера (бот + веб)
- Integram CRM (76 товаров, 285+ клиентов, 326+ заказов)

## 2. Сильные стороны

### 2.1 Функциональные

| Аспект | Описание |
|--------|----------|
| **Гибридный поиск** | Уникальная комбинация семантика 70% + стилометрия 30% + keyword-буст обеспечивает точные ответы в стиле автора |
| **Многоагентная архитектура** | LangGraph-оркестратор с 6 типами интентов (consult, order, delivery, status, stats, chitchat) |
| **CRM-интеграция** | Полный CRUD для товаров, заказов, клиентов через Integram API |
| **PWA-терминалы** | Offline-first подход для работы на пасеке с плохим интернетом |
| **Веб-панель** | Дашборд с графиками, журнал заказов, управление складом |
| **FSM заказов** | 7-шаговый диалог оформления заказа с авто-подстановкой данных клиента |

### 2.2 Архитектурные

| Аспект | Описание |
|--------|----------|
| **Модульность** | Агенты (beebot, logist, analyst) изолированы в отдельные модули |
| **Инфраструктура** | Docker + systemd + SSH-туннель — надёжная схема обхода блокировок Groq |
| **Retry-логика** | 3 попытки с exponential backoff на всех внешних вызовах |
| **Pydantic-модели** | Валидация данных на границе API |
| **Чистое разделение** | Frontend (Vue/Vite) и Backend (FastAPI) в отдельных контейнерах |

## 3. Критические проблемы

### 3.1 Безопасность

#### JWT-секрет по умолчанию
**Файл:** `src/web/api.py`
**Проблема:** `SECRET_KEY = os.getenv("JWT_SECRET", "dev-secret-change-in-production")` — если переменная окружения не установлена, используется предсказуемый ключ. Любой может сгенерировать валидный JWT-токен.
**Решение:** Убрать значение по умолчанию, сделать обязательной переменной. При отсутствии — не запускать сервер.

#### Открытый CORS
**Файл:** `src/web/api.py`
**Проблема:** `allow_origins=["*"]` — разрешены запросы с любого домена.
**Решение:** Ограничить до конкретных доменов (VPS IP, localhost для разработки).

### 3.2 Нерабочий код

#### UDS-интеграция
**Файл:** `src/integrations/uds.py`, строка ~349
**Проблема:** Вызывает `self._request()`, но такого метода нет в классе. Используется `self.http.get/post` в остальных методах. UDS-интеграция полностью нерабочая.
**Решение:** Реализовать метод `_request()` или заменить вызовы на `self.http`.

#### Логист без CRM
**Файл:** `src/agents/logist.py`, строки ~170-172
**Проблема:** В финальном шаге `create_order()` агента логиста нет реальной записи в CRM. Заказ создаётся только как текст уведомления, но не сохраняется в Integram.
**Решение:** Добавить вызов `integram_api.create_order()` при подтверждении заказа.

### 3.3 Баг маршрутизации (ИСПРАВЛЕН)

#### Router: `/orders/new` после `/orders/:id`
**Файл:** `web/src/router/index.js`
**Проблема:** Маршрут `/orders/new` объявлен после `/orders/:id`, поэтому Vue Router интерпретирует `new` как `:id` параметр, и страница создания заказа никогда не открывается.
**Статус:** Исправлено в этой сессии.

## 4. Серьёзные проблемы

### 4.1 Дублирование кода

#### Lookup-константы в 3 местах
Одни и те же ID справочников CRM (статусы, источники, категории) определены в:
1. `src/integram_api.py` — `STATUS_IDS`, `SOURCE_IDS`
2. `src/integram_client.py` — `STATUS_LOOKUP`, `SOURCE_LOOKUP`
3. `src/web/api.py` — `CATEGORY_IDS`, `STATUS_IDS`

**Риск:** При добавлении нового значения в CRM нужно менять в 3 файлах. Рассинхронизация уже есть — некоторые ID отличаются.
**Решение:** Вынести в единый модуль `src/crm_constants.py`.

#### Двойная инициализация CRM-сессии
**Файл:** `src/bot.py`, строки ~61 и ~892
**Проблема:** `setup_admin()` вызывается дважды — при импорте и в `on_startup`. Повторная аутентификация может сбросить активную сессию.
**Решение:** Убрать вызов при импорте, оставить только в `on_startup`.

### 4.2 Алгоритмические ошибки

#### FAISS: порядок нормализации
**Файл:** `src/knowledge_base.py`, строка ~148
**Проблема:** Нормализация эмбеддингов после добавления в индекс. При `IndexFlatIP` (inner product = cosine similarity для нормализованных векторов) нормализация должна быть **до** добавления.
**Влияние:** Качество поиска может быть ниже оптимального, но работает приемлемо благодаря keyword-бусту.

#### Деление на ноль в аналитике
**Файл:** `src/agents/analyst.py`
**Проблема:** При расчёте среднего чека, если заказов 0, произойдёт деление на ноль.
**Решение:** Добавить проверку `if count > 0`.

### 4.3 Race conditions

#### Таймаут FSM-диалога
**Файл:** `src/bot.py`, строки ~479-508
**Проблема:** Словарь `_order_timeouts` модифицируется из разных корутин без блокировок. При конкурентных заказах от разных пользователей возможна потеря данных.
**Решение:** Использовать `asyncio.Lock` или `collections.defaultdict` с атомарными операциями.

### 4.4 Отсутствие обработки ошибок

#### Калькулятор доставки
**Файлы:** `src/delivery/cdek.py`, `src/delivery/pochta.py`
**Проблема:** При ошибке API (таймаут, 500) функция возвращает `0` вместо индикации ошибки. Клиент видит «Доставка: 0₽» и думает, что доставка бесплатна.
**Решение:** Возвращать `None` при ошибке, показывать «Не удалось рассчитать».

## 6. Конфликты и расхождения логик

### 6.1 Два CRM-клиента

Существуют два модуля для работы с CRM:
- `src/integram_api.py` — низкоуровневый HTTP-клиент (используется ботом и веб-панелью)
- `src/integram_client.py` — высокоуровневая обёртка (используется оркестратором)

**Проблема:** Дублируют функционал, имеют разные маппинги полей, могут использовать разные сессии. При изменении API нужно править оба файла.

### 6.2 Модели данных

- `src/models.py` — Pydantic-модели для бота/оркестратора
- `src/web/api.py` — отдельные Pydantic-модели для веб-API

**Проблема:** Модель `Order` в `models.py` и структура ответа `/api/orders` имеют разные поля и naming conventions.

### 6.3 Статусы заказов

Статусы определены как lookup-значения CRM, но маппинг статус→ID отличается между модулями:
- `integram_api.py`: использует `STATUS_IDS` dict
- `integram_client.py`: использует `STATUS_LOOKUP` dict
- `web/api.py`: использует свой набор

Добавление нового статуса требует правки в 3+ местах.

### 6.4 Обработка дат

- Бот: даты в формате `dd.mm.yyyy`
- Веб-API: парсит через regex с фильтром 2024-2030
- CRM: хранит как строки, иногда с мусором (например `31.03.2500`)

Нет единого модуля для работы с датами из CRM.
