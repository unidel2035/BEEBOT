## Git и деплой

### AP-001: git pull на VPS после squash-мержа
**Категория:** Git / Deploy
**Серьёзность:** КРИТИЧНО
**Правило:** НЕ использовать `git pull` на VPS
**Почему:** squash-мерж создаёт новый коммит — история расходится, pull создаёт merge commit или конфликт
**Вместо этого:** всегда `git fetch origin main && git reset --hard origin/main`

### AP-002: git add -A или git add .
**Категория:** Git
**Серьёзность:** ВАЖНО
**Правило:** НЕ использовать `git add -A` или `git add .`
**Почему:** риск случайно закоммитить `.env`, секреты, большие бинарные файлы, временные файлы
**Вместо этого:** `git add src/specific_file.py tests/test_specific.py` — только нужные файлы

### AP-003: push напрямую в upstream
**Категория:** Git
**Серьёзность:** КРИТИЧНО
**Правило:** НЕ пушить напрямую в `alekseymavai/BEEBOT`
**Почему:** нет прав на прямой push, только через fork
**Вместо этого:** push в `unidel2035/BEEBOT` → PR → squash мерж. Переключать: `gh auth switch -u unidel2035 && gh auth setup-git`; после деплоя: `gh auth switch -u gaveron18`

### AP-004: амендить коммит после хука
**Категория:** Git
**Серьёзность:** ВАЖНО
**Правило:** НЕ использовать `git commit --amend` после падения pre-commit хука
**Почему:** при падении хука коммит НЕ создался — amend изменит предыдущий коммит
**Вместо этого:** исправить, `git add`, создать НОВЫЙ коммит

## Инфраструктура

### AP-005: изменение .env на VPS без явного указания
**Категория:** Infra
**Серьёзность:** КРИТИЧНО
**Правило:** НЕ трогать `.env` на VPS без явного разрешения Андрея
**Почему:** `.env` содержит все секреты продакшна; ошибка = бот упал для реальных пользователей
**Вместо этого:** предложить изменение, подождать подтверждения, применять точечно через `sed -i`

### AP-006: `network_mode: bridge` в docker-compose
**Категория:** Infra
**Серьёзность:** КРИТИЧНО
**Правило:** НЕ убирать `network_mode: host` из docker-compose
**Почему:** бот использует SSH-туннели через localhost (Groq :8990, Telegram SOCKS5 :9150) — без host mode они недоступны
**Вместо этого:** всегда оставлять `network_mode: host`

### AP-007: SSH-алиас beebot-vps
**Категория:** Infra
**Серьёзность:** ВНИМАНИЕ
**Правило:** НЕ использовать SSH-алиас `beebot-vps`
**Почему:** алиас не работает (DNS не разрешается)
**Вместо этого:** прямой IP `ssh ai-agent@185.233.200.13`

## Тестирование

### AP-008: мокировать базу данных в тестах
**Категория:** Testing
**Серьёзность:** ВАЖНО
**Правило:** НЕ мокировать SQLite/базу данных в интеграционных тестах
**Почему:** мок-тесты могут проходить пока продакшн-миграция сломана (был реальный инцидент в проекте)
**Вместо этого:** использовать реальную in-memory SQLite (`:memory:`) или тестовую БД

### AP-009: пропускать тест "потому что и так очевидно"
**Категория:** Testing
**Серьёзность:** ВАЖНО
**Правило:** НЕ пропускать написание теста для "очевидного" кода
**Почему:** у нас test-first подход — тест документирует ожидаемое поведение
**Вместо этого:** написать хотя бы один тест на happy path + один на edge case

## Код

### AP-010: трогать код вне scope задачи
**Категория:** Code
**Серьёзность:** ВАЖНО
**Правило:** НЕ исправлять "заодно" другой код который заметил по дороге
**Почему:** размывает scope PR, создаёт неожиданные изменения, сложнее ревьюить
**Вместо этого:** создать задачу в plan.md, выполнить отдельным PR

### AP-011: угадывать содержимое файла без Read
**Категория:** Code
**Серьёзность:** КРИТИЧНО
**Правило:** НЕ редактировать файл не прочитав его полностью (Read перед Edit)
**Почему:** структура файлов часто отличается от ожидаемой; Edit без Read = гарантированный баг
**Вместо этого:** ВСЕГДА Read → затем Edit

### AP-012: дублировать логику агентов
**Категория:** Code
**Серьёзность:** ВАЖНО
**Правило:** НЕ писать одинаковую бизнес-логику в двух агентах
**Почему:** был реальный случай: `_get_crm_context` + `_build_context_from_snapshot` ~330 строк идентичного кода в admin_chat.py
**Вместо этого:** выносить в Service layer (`src/services/`), агенты только оркестрируют
