# BEEBOT — Архитектурные решения (ADR)

> Здесь фиксируется ПОЧЕМУ выбрали X а не Y.
> Каждый ADR = принятое решение + причина + альтернативы которые отклонили.

---

## Формат записи

```
## ADR-NNN: Название
```

## Инфраструктура

### ADR-001: SSH-туннель вместо прямых Groq API-запросов с VPS
**Статус:** Принято
**Дата:** 2026-03-01
**Контекст:** Groq API блокирует IP-адреса VPS-провайдеров (OVH, Hetzner и т.д.)
**Решение:** groq-proxy.py на hive (порт 8990) + SSH-туннель от VPS до hive
**Причина:** Groq не предоставляет whitelist IP; VPS-IP в блоке; hive — домашний IP, не в блоке
**Альтернативы:**
- OpenAI API — стоит дороже, тоже может блокировать VPS IP
- Selfhosted LLM — требует GPU, недоступно на VPS 2 GB RAM
- Другой провайдер LLM — та же проблема с блокировкой VPS IP
**Последствия:** hive — Single Point of Failure для всего LLM-стека бота

### ADR-002: SOCKS5-прокси для Telegram API
**Статус:** Принято
**Дата:** 2026-03-01
**Контекст:** Telegram API блокирует IP VPS (аналогично Groq)
**Решение:** tg_socks_proxy.py на hive (порт 9150) + SOCKS5 в aiogram config
**Причина:** Telegram официально поддерживает proxy в aiogram; SOCKS5 — минимальный оверхед
**Альтернативы:**
- MTProto-прокси — сложнее в настройке, избыточно для нашего случая
- VPN на VPS — меняет весь трафик, влияет на другие сервисы
**Последствия:** hive — SPOF для Telegram тоже. Оба канала (Groq + Telegram) зависят от hive

### ADR-003: network_mode: host в docker-compose
**Статус:** Принято
**Дата:** 2026-03-01
**Контекст:** Бот внутри Docker-контейнера должен дотянуться до SSH-туннелей на localhost
**Решение:** `network_mode: host` для beebot-контейнера
**Причина:** В bridge-режиме localhost контейнера ≠ localhost хоста; host — прямой доступ
**Альтернативы:**
- `extra_hosts: host.docker.internal` — работает только на Linux с явным маппингом, не всегда надёжно
- host.docker.internal — не стандартизировано на Linux
**Последствия:** Нет изоляции сети контейнера. Приемлемо для single-tenant VPS

## Git и деплой

### ADR-004: squash-мерж вместо обычного мержа
**Статус:** Принято
**Дата:** 2026-03-10
**Контекст:** Upstream alekseymavai/BEEBOT управляется не нами. Нужен чистый git-лог.
**Решение:** Squash-мерж всех PR в upstream
**Причина:** Чистая история в main; один коммит = одна задача; легче revert
**Альтернативы:**
- Rebase мерж — сохраняет все коммиты, но засоряет историю при активной разработке
- Обычный merge commit — создаёт лишние merge-коммиты
**Последствия:** После squash-мержа VPS нельзя делать `git pull` — только `git reset --hard origin/main`

### ADR-005: fork workflow через unidel2035
**Статус:** Принято
**Дата:** 2026-03-01
**Контекст:** Нет прав на прямой push в alekseymavai/BEEBOT
**Решение:** Fork unidel2035/BEEBOT → PR → squash мерж в upstream → VPS reset
**Причина:** Стандартная практика для репозиториев без прав на прямой push
**Альтернативы:**
- Запросить права на push — зависит от alekseymavai, не наш контроль
**Последствия:** Двойная смена gh auth (unidel2035 → gaveron18) при каждом деплое

## Архитектура кода

### ADR-006: CRM v1/v2 через фабрику (crm_factory.py)
**Статус:** Принято
**Дата:** 2026-03-15
**Контекст:** Переход с Integram v1 на v2 API при наличии 1915 живых заказов в production
**Решение:** `crm_factory.py` + `INTEGRAM_V2=true/false` env-флаг; оба клиента за единым интерфейсом
**Причина:** Нельзя рискнуть сломать production одним переключением; нужен постепенный переход
**Альтернативы:**
- Одновременный переход — высокий риск; если что-то сломается → потеря заказов
- Параллельная запись в v1 и v2 — двойная сложность, риск дублей
**Последствия:** `crm_constants.py` — единый источник всех CRM ID; оба клиента должны его использовать

### ADR-007: LangGraph checkpointer на SQLite (data/checkpoints.db)
**Статус:** Принято
**Дата:** 2026-04-01
**Контекст:** LangGraph требует checkpointer для персистентности диалоговых состояний (M1)
**Решение:** SqliteSaver в `data/checkpoints.db`; в production — Redis 7
**Причина:** SQLite — zero-dependency; Redis в docker-compose для production (меньше RAM-overhead)
**Альтернативы:**
- Только in-memory — теряем состояние при рестарте бота
- PostgreSQL — избыточно для нашей нагрузки (сотни, не миллионы пользователей)
**Последствия:** data/checkpoints.db в .gitignore; при деплое — пустая БД (первый запуск пересоздаёт)

### ADR-008: MemoryService как единый API памяти (M4)
**Статус:** Принято
**Дата:** 2026-04-04
**Контекст:** UserMemory (SQLite) использовалась напрямую в нескольких агентах → дублирование
**Решение:** `src/memory_service.py` — единая точка входа для всей памяти агентов
**Причина:** Service layer; агенты не должны знать о деталях хранения (SQLite vs Redis vs future)
**Альтернативы:**
- Оставить прямой доступ к UserMemory — работает, но нарушает принцип DI
**Последствия:** Все агенты используют MemoryService; прямой доступ к UserMemory — антипаттерн

### ADR-009: EventBus в lifespan (src/bus.py)
**Статус:** Принято
**Дата:** 2026-03-20
**Контекст:** Нужна асинхронная связь между компонентами (уведомления, обновления статуса)
**Решение:** In-process EventBus через asyncio.Queue; регистрация в startup.py lifespan
**Причина:** Нет необходимости в внешнем брокере (RabbitMQ, Kafka) при нашей нагрузке
**Альтернативы:**
- Redis Pub/Sub — оверхед, дополнительная зависимость
- Прямые вызовы между модулями — жёсткая связность
**Последствия:** AgentBus только in-process; никакого AGENT_BUS_URL (см. antipatterns)

### ADR-010: Гибридный поиск в KB (70% семантика + 30% стилометрия)
**Статус:** Принято
**Дата:** 2026-03-05
**Контекст:** Только семантический поиск давал нерелевантные результаты при специфических вопросах
**Решение:** 70% FAISS cosine similarity + 30% стилометрия + keyword-буст из CRM-каталога
**Причина:** Стилометрия помогает фильтровать по "голосу автора"; keyword-буст улучшает поиск товаров
**Альтернативы:**
- Только BM25 — не учитывает семантическую близость
- 100% семантика — теряем точность на специфических терминах
**Последствия:** 276 чанков; пересборка KB: `docker exec beebot python -m src.build_kb`
