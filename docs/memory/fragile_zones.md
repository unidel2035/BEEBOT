# BEEBOT — Хрупкие зоны кода

> Файлы и области с повышенным риском при изменении.
> Читать перед тем как трогать что-либо из этого списка.
> **Правило:** любое изменение хрупкой зоны → обязательный тест перед деплоем.

---

## Уровни хрупкости

- 🔴 **КРИТИЧНО** — изменение с высокой вероятностью ломает production
- 🟡 **ОСТОРОЖНО** — нужно понимать все зависимости перед правкой
- 🟢 **ВНИМАНИЕ** — править можно, но читать внимательно

---

## Инфраструктура

### 🔴 docker-compose.yml — network_mode: host
**Файл:** `docker-compose.yml`
**Риск:** Удаление или изменение `network_mode: host` → бот не дотянется до SSH-туннелей
**Симптомы:** `Connection refused` к Groq (:8990) и Telegram SOCKS5 (:9150)
**Правило:** Не трогать network_mode. Не добавлять `networks:` без явного указания Андрея.

### 🔴 .env на VPS — секреты и флаги
**Файл:** `/home/ai-agent/BEEBOT/.env`
**Риск:** Ошибка в .env → бот не запускается, потеря заказов в production
**Правило:** Не трогать без явного разрешения Андрея. Изменения только точечно через `sed -i`.
Перед применением — проверить текущее значение: `grep KEY .env`

### 🔴 src/startup.py — единая точка инициализации
**Файл:** `src/startup.py` (~335 строк)
**Риск:** ImportError в startup.py = бот не стартует вообще
**Зависимости:** Импортирует `src/services/*`, `src/memory_service.py`, `src/orchestrator.py`
**Симптомы:** `docker logs beebot` показывает ImportError в первых строках
**Правило:** При удалении любого сервиса — сначала убрать из startup.py. При добавлении — добавить в startup.py.

## CRM и данные

### 🔴 src/crm_constants.py — единый источник всех CRM ID
**Файл:** `src/crm_constants.py`
**Риск:** Изменение любого ID → все запросы к CRM ломаются, данные записываются не в те поля
**Зависимости:** Используется в `integram_client.py`, `integram_v2_client.py`, всех агентах, роутерах
**Правило:** Не менять числовые константы без проверки через Integram UI (ai2o.ru/bibot).
Добавлять — можно. Переименовывать переменные — только с grep по всему проекту.

### 🔴 src/crm_factory.py — переключатель CRM v1/v2
**Файл:** `src/crm_factory.py`
**Риск:** Ошибка → все компоненты получат неправильный CRM-клиент
**Флаг:** `INTEGRAM_V2=true/false` в .env; **на VPS сейчас: false (v1)**
**Правило:** Переключение INTEGRAM_V2=true на VPS — отдельная задача Fix-2, требует явного ОК Андрея.

### 🟡 data/checkpoints.db — LangGraph состояния
**Файл:** `data/checkpoints.db` (+ `-shm`, `-wal`)
**Риск:** Удаление → потеря всех активных диалоговых состояний пользователей
**Правило:** Не удалять в production. Файл в .gitignore — не коммитить.

### 🟡 data/processed/ — FAISS-индекс и чанки
**Файлы:** `data/processed/index.faiss`, `data/processed/chunks.json`
**Риск:** Удаление или перезапись → KB сломана, бот не может консультировать
**Правило:** Пересборка только через `docker exec beebot python -m src.build_kb`.
Не трогать вручную. После пересборки — проверить что `len(chunks) == 276`.

## Агенты и оркестратор

### 🟡 src/orchestrator.py — классификация интентов
**Файл:** `src/orchestrator.py`
**Риск:** Изменение классификации → агент перестаёт отвечать на нужный тип запросов
**Интенты:** consult / order / edit / track / stats / greeting
**Правило:** Любое изменение промпта классификации → тест на всех 6 интентах.
Добавление нового интента → добавить в StateGraph + тест.

### 🟡 src/agents/logist.py — 7-шаговый FSM заказа
**Файл:** `src/agents/logist.py`
**Риск:** Изменение порядка или условий FSM → заказы застревают или создаются с неполными данными
**Шаги:** товары → ФИО → телефон → адрес → доставка → подтверждение → CRM
**Правило:** Test-first обязателен. Проверять весь happy path целиком, не только изменённый шаг.

### 🟡 src/memory_service.py — единый API памяти (M4)
**Файл:** `src/memory_service.py`
**Риск:** Изменение сигнатур методов → все агенты, которые его используют, ломаются
**Зависимости:** orchestrator.py, admin_chat.py (после M5), возможно другие
**Правило:** Изменение публичного API MemoryService = breaking change → инкрементировать версию или создать новый метод.

## Веб-панель

### 🟡 src/web/routers/sse.py — Server-Sent Events
**Файл:** `src/web/routers/sse.py`
**Риск:** Ошибка → веб-панель не получает real-time обновления, зависает
**Правило:** Не блокировать event loop в SSE-генераторе. Тестировать через браузер, не только pytest.

### 🟡 web/src/stores/offline.js — IndexedDB + sync queue
**Файл:** `web/src/stores/offline.js`
**Риск:** Изменение схемы IndexedDB без миграции → данные PWA на устройстве недоступны
**Правило:** При изменении структуры хранилища — инкрементировать DB_VERSION и писать миграцию.

### 🟢 src/web/routers/auth.py — JWT авторизация
**Файл:** `src/web/routers/auth.py`
**Риск:** Ошибка → веб-панель недоступна (все маршруты закрыты)
**Правило:** Не менять алгоритм JWT без обновления всех клиентов. Проверять что `/api/auth/login` работает после изменений.

## Integrations

### 🔴 src/integrations/uds.py — UDS поллер
**Файл:** `src/integrations/uds.py`
**Риск:** Ошибка → UDS-заказы перестают синхронизироваться; продажи не отражаются в CRM
**Особенности:**
- Дедупликация по UDS-номеру
- Catch-up с 01.01.2024 (при первом запуске — долго!)
- Источник: req 1076 = 1083 ("UDS") — не менять (ADR)
**Правило:** Не трогать логику дедупликации без понимания схемы Integram.

### 🟡 src/delivery/cdek.py — СДЭК API v2
**Файл:** `src/delivery/cdek.py`
**Риск:** Изменение OAuth2-логики → все СДЭК-расчёты и трекинг ломаются
**Правило:** Тестировать с реальным СДЭК API (sandbox), не мокировать OAuth.

## hive-сервисы (не в репозитории)

### 🔴 groq-proxy.service + groq-tunnel.service
**Расположение:** systemd на hive
**Риск:** Падение сервиса → весь LLM-стек бота перестаёт работать
**Симптом:** Бот не отвечает на сообщения; `docker logs beebot` показывает timeout к :8990
**Мониторинг:** `systemctl status groq-proxy groq-tunnel`

### 🔴 tg-socks.service
**Расположение:** systemd на hive
**Риск:** Падение → бот не может подключиться к Telegram API
**Симптом:** `ConnectionError` в docker logs beebot
**Мониторинг:** `systemctl status tg-socks`

## Правила работы с хрупкими зонами

1. **Перед правкой** — прочитать этот файл + прочитать сам файл целиком (Read, не Edit вслепую)
2. **Тест обязателен** — для 🔴 зон test-first, без исключений
3. **После правки** — проверить логи бота (`docker logs --tail 50 beebot`)
4. **Если сломалось** — первая гипотеза: я сломал. Не искать внешние причины пока не исключены мои действия.
5. **Обновить этот файл** — если обнаружил новую хрупкую зону
