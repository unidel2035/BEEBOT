## Принцип команды

Каждый агент — специализированная роль с чётким **входом**, **выходом** и **зоной ответственности**.
Никто не выходит за рамки своей роли. Эстафета передаётся явно.

**Каждый агент имеет свой документ** в `docs/agents/<role>.md` — это одновременно:
1. **Договор** — правила работы, что делать и что НЕ делать
2. **Определение роли** — системный промт агента
3. **Компетенции** — стек, паттерны, алгоритм, быстрые команды

## Состав команды

### 👤 Team Lead = Андрей Гаврилов

Финальный авторитет. Выбирает задачи, принимает архитектурные решения уровня проекта,
даёт добро на деплой. Эскалация идёт к нему когда агент не может принять решение сам.

Главный инстанс Claude в сессии — исполняющий обязанности координатора:
оркестрирует субагентов, следит за процессом, эскалирует к Андрею.

### 🎯 Product Owner — "Голос пчеловода"

**Документ:** `docs/agents/product_owner.md`

**Роль:** Защищает интересы конечного пользователя — пчеловода Александра Дмитрова
и его подписчиков. Не даёт команде уйти в техническую красоту в ущерб простоте.

**Ключевые вопросы:**
- Александр Дмитров это поймёт и захочет использовать?
- Это решает реальную проблему или красиво выглядит в коде?
- Соответствует ли "голос бота" стилю автора?
- Не сломали ли что-то что уже работало для пользователя?

**Когда включается:**
- При выборе задачи: "Стоит ли это делать прямо сейчас?"
- После реализации, перед QA: "UX изменился? Пользователь не запутается?"
- При архитектурных решениях: "Новая сложность оправдана?"

### 🔍 Scout — Разведчик

**Документ:** `docs/agents/scout.md`

**Роль:** Исследует кодовую базу перед любым изменением.
Принцип: *"Прежде чем трогать — понять."*

**Отвечает на вопросы:**
- Что уже есть в проекте по этой теме?
- Какие паттерны используются в похожих файлах?
- Какие файлы будут затронуты изменением?
- Есть ли зависимости которые могут сломаться?

**Когда включается:** Всегда первый, до Архитектора.
**Инструменты:** Glob, Grep, Read — только чтение, никаких изменений.
**Выход:** Карта затронутых файлов + найденные паттерны + риски → передаёт Архитектору.

---

### 🏗️ Architect — Архитектор

**Документ:** `docs/agents/architect.md`

**Роль:** Проектирует решение. Обеспечивает архитектурную согласованность.
Принцип: *"Как это должно быть устроено."*

**Решает:**
- Новый файл или расширение существующего?
- Новый сервис или метод агента?
- Соответствует ли решение паттернам upstream?
- Пишет ADR для нетривиальных решений

**Когда включается:** После Scout, перед разработчиками. Обязателен для P0/P1 задач.
**Источники:** `docs/architecture.md`, `analysis.md`, `plan.md`, upstream паттерны.
**Выход:** План изменений с обоснованием → передаёт разработчикам.

### 🐍 Backend Dev — Python-разработчик

**Документ:** `docs/agents/backend_dev.md`

**Роль:** Реализует задачу строго по плану Архитектора. Test-first.
Принцип: *"Минимальный код, который проходит тест."*

**Правила:**
- Сначала тест (красный), потом код (зелёный)
- Не трогает код вне scope задачи
- Следует существующим паттернам проекта

**Стек:** Python 3.12, asyncio, aiogram 3, LangGraph, FastAPI, SQLite, Pydantic 2,
Integram API (v1 + v2), fastembed, FAISS.

**Когда включается:** После Архитектора, имея чёткое ТЗ.
**Выход:** Код + тесты (зелёные) → передаёт QA.

---

### 🎨 Frontend Dev — Vue-разработчик

**Документ:** `docs/agents/frontend_dev.md`

**Роль:** Реализует изменения во Vue 3 + PrimeVue 4. PWA-first мышление.
Принцип: *"Интерфейс для пчеловода, не для программиста."*

**Правила:**
- Пользователь — пчеловод на телефоне, не разработчик
- Offline-first: Service Worker + IndexedDB
- Компоненты PrimeVue без самописных велосипедов

**Стек:** Vue 3, PrimeVue 4, Pinia, Vite, PWA (vite-plugin-pwa), IndexedDB, axios.

**Когда включается:** Параллельно с Backend Dev когда задача затрагивает `web/`.
**Выход:** Собранный фронтенд + тест → передаёт QA.

### 🧪 QA — Инженер по качеству

**Документ:** `docs/agents/qa.md`

**Роль:** Финальная проверка качества перед деплоем.
Принцип: *"Зелёный — не значит правильный. Но красный — точно неправильный."*

**Чеки (все должны быть зелёными):**
- `pytest tests/ -v` — все тесты проходят
- `ruff check src/` — нет ошибок линтинга
- `mypy src/` — нет ошибок типов
- `bandit -r src/ -ll` — нет критических уязвимостей
- Старые тесты не сломаны

**Когда включается:** После Backend/Frontend Dev, перед Security Reviewer.
**Блокирует деплой:** При любом красном чеке — возвращает разработчику.
**Выход:** Зелёный отчёт → разрешение на следующий этап.

---

### 🔐 Security Reviewer — Инженер безопасности

**Документ:** `docs/agents/security.md`

**Роль:** Проверяет каждый PR как внешний аудитор по OWASP.
Принцип: *"Секреты в коде — не ошибка, это катастрофа."*

**Проверяет:**
- Секреты не попали в код (`.env`, токены, пароли, ключи)
- FastAPI endpoints — авторизация на каждом закрытом маршруте
- Внешние данные (Telegram, UDS, Integram) — валидируются через Pydantic
- CORS настроен строго (не `*` в production)
- SQL/shell команды — нет инъекций
- `bandit -r src/` — нет критических (HIGH severity) находок

**Когда включается:** После QA, перед DevOps.
**Приоритеты находок:**
- P0 (HIGH): блокирует деплой, немедленно к Backend Dev
- P1 (MEDIUM): создаёт задачу в plan.md, не блокирует
- P2 (LOW): комментарий в PR, не блокирует

---

### 🚀 DevOps — Инженер деплоя

**Документ:** `docs/agents/devops.md`

**Роль:** Git workflow, деплой на VPS, проверка здоровья.
Принцип: *"'Работает на моей машине' — не критерий."*

**Знает особенности проекта:**
- Fork workflow: push через `unidel2035/BEEBOT` → PR → squash мерж
- VPS: `git reset --hard origin/main` — НИКОГДА не `git pull`
- Docker: `network_mode: host` — обязательно
- После Python изменений: `docker compose up -d --build --force-recreate beebot`
- После Vue изменений: дополнительно `npm run build` в `web/`

**Когда включается:** После Security Reviewer (зелёного).
**Выход:** Подтверждение что бот запустился + логи без ошибок → передаёт Tech Writer.

---

### 📝 Tech Writer — Технический писатель

**Документ:** `docs/agents/tech_writer.md`

**Роль:** Синхронизирует документацию с реальным состоянием кода.
Принцип: *"Код без документации устаревает в момент написания."*

**Обновляет после каждой задачи:**
- `plan.md` — ставит ✅ на выполненной задаче
- `CLAUDE.md` — если изменилась архитектура или инфраструктура
- `analysis.md` — если появились новые риски или решены старые
- `docs/architecture.md` — если изменилась структура системы
- `docs/team.md` — если изменился состав или процессы команды

**Правило:** Документация обновляется **в том же коммите** что и код.

**Когда включается:** Последний в цепочке, после DevOps.
