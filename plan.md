- Таблица **История статусов** (6156) в Integram: REF-заказ + Статус из + Статус в + Дата + Комментарий

- Таблица **Партии отправки** (6404): группировка заказов для массовой отправки
- Чеклист-поля заказа: «Адрес СДЭК уточнён», «Клиент оповещён», «Наличие проверено»

### ✅ 4.1 Расщепление api.py — выполнено

`src/web/api.py` разбит на 167 строк (main router) + `src/web/routers/` (auth, orders, clients, products, dashboard, export, users, sse).

### ✅ 4.4 Партии отправки — выполнено (25.03.2026)

- `BatchesView.vue` — страница «Партии отправки»: таблица, создание, просмотр заказов партии
- `OrderDetailView.vue` — Select «Партия» с авто-сохранением

- Бот-команда `/batch` — **отложена** (P3, нет запроса от пчеловода)

### ✅ 4.5 История статусов в веб-панели — выполнено (25.03.2026)

- `GET /api/orders/{id}/history` + `integram_client.get_order_history()`
- `OrderDetailView.vue`: timeline статусов (дата + переход + комментарий авто-трекера)
- Обновляется после каждой смены статуса на странице

### ✅ 4.6 Чеклист заказа в веб-панели — выполнено (25.03.2026)

- `PATCH /api/orders/{id}/checklist` + `integram_client.update_order_checklist()`
- `OrderDetailView.vue`: три checkbox «Наличие / Адрес СДЭК / Клиент оповещён»

### ✅ Reference-поля Integram — исправлено (27.03.2026)

**Проблема:** Integram `_m_save` игнорирует обновления reference-полей (статус, источник, доставка).

**Решение:** `integram_api.set_reference_field()` — `_m_del` (все старые links) + `_m_set` (новая link). Применено в `update_order_status()` и `update_order()`. Все веб-панельные и TG-смены статуса теперь работают корректно.

Вручную исправлено: 7 исторических заказов в статусе «Новый» (id: 4207–4735) и заказ 4020 с 12 spurious-записями.

### ✅ 4.7 Расширение онтологии — выполнено (27.03.2026)

- Добавлено **62 симптома** (ID 6717–6839) с ключевыми словами — было 12, стало 74
- Добавлено **77 показаний к применению** (ID 6841–7191) — все с типом (Основное / Вспомогательное / С осторожностью) и дозировкой
- Охват: ОРВИ/иммунитет, ЖКТ, суставы, нервная система, сердце/сосуды, кожа, мужское/женское здоровье, обмен веществ
- `OntologyCache` подхватывает данные автоматически при следующем рестарте бота

### ✅ 4.2 Объединение дублей клиентов — выполнено (25.03.2026)

- `GET /api/clients/duplicates` — группы по одинаковому телефону
- `POST /api/clients/merge` — перенос заказов + удаление дубля + копирование telegram_id
- `ClientsView.vue` — кнопка «Дубли (N)» + диалог объединения с подтверждением
- UDS-автосвязывание — **отложено** (дублей по телефону всего 2, UDS поллер имеет свою логику)

### ✅ 4.3 Объединение констант CRM — выполнено (25.03.2026)

- Все `TABLE_*` и `REQ_*` перенесены из `integram_api.py` в `crm_constants.py`
- `integram_api.py` re-export'ит для обратной совместимости
- Единый источник: `crm_constants.py` для всех CRM ID и списков

## Фаза 6: Масштабирование и Монетизация

### 6.1 Перенос LLM-инфраструктуры (P3)

- [ ] Оценить возможность запуска Ollama на VPS (4 GB RAM minimum)
- [ ] Fallback-цепочка: Groq API → Ollama на VPS → упрощённый ответ
- [ ] Мониторинг доступности hive-туннеля с авто-fallback

### 6.2 Аналитика и отчётность (P3)

- [ ] Экспорт PDF-отчётов (выручка за период, топ товаров, клиентский сегмент)
- [ ] Прогноз спроса на основе истории заказов (LLM + статистика)
- [ ] Алерты при низком складском остатке → авто-уведомление пчеловоду

### 6.3 Платёжная интеграция (P3)

- [ ] YooKassa или Тинькофф для онлайн-оплаты заказов
- [ ] QR-код для оплаты в Telegram
- [ ] Автоматическая смена статуса после подтверждения оплаты

### 6.4 Инфраструктура (P3)

- [ ] Апгрейд VPS до 4 GB RAM (снять зависимость от swap)
- [ ] Перенести Groq-прокси на VPS (устранить SPOF от hive)
- [ ] Nginx reverse proxy с SSL для веб-панели (https://)
- [ ] Резервное копирование данных CRM
